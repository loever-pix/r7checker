# Ranked Charm Scrape Design

## Goal

Populate every Rainbow Six Siege ranked charm image from Ranked 1.0 through Ranked 3.0 using `r6.skin/search/`, then make those images available anywhere the app renders or reports ranked charms.

## Current Context

The app already synthesizes ranked charms in `lib/player.js` from `seasonRanks`. `buildRankedCharms()` creates one charm per earned rank tier for each ranked season, and both profile checking and bulk checking reuse that player data. The current image lookup uses ranked charm entries from `lib/skins_cache.json`, falling back to rank badge art when catalog art is missing.

`r6.skin/search/` is a static search UI backed by the Meilisearch API at `https://search.r6s.skin/indexes/skins/search`. Its search script sends POST requests with a public bearer key and receives hits containing names and image URLs.

## Approach

Create a separate generated data file, `data/ranked-charm-images.json`, instead of modifying `lib/skins_cache.json`. The generated file will be keyed by normalized tier and season name, for example `platinum|silent hunt`, and each value will include at least the charm image URL and the source item name. Keeping this as a separate file makes regeneration easy and avoids noisy changes to the large catalog.

Add a repeatable scraper script that queries the `r6.skin` search API by operation name plus rank tier. The script will iterate every known season in `lib/player.js` from Black Ice through current Ranked 3.0 seasons and every eligible ranked tier for that season. It will conservatively choose hits whose name and metadata indicate a ranked charm for the requested tier and operation.

Update `lib/player.js` so `rankedCharmImage(tier, seasonName)` checks the generated `data/ranked-charm-images.json` first, then falls back to the existing `lib/skins_cache.json` ranked charm lookup, then uses rank badge art as it does today.

## Data Shape

`data/ranked-charm-images.json` will be generated as:

```json
{
  "_comment": "Generated by scripts/scrape-ranked-charms.js from r6.skin search results.",
  "_lastUpdated": "2026-06-15",
  "items": {
    "emerald|silent hunt": {
      "name": "Emerald (Silent Hunt)",
      "tier": "emerald",
      "season": "Silent Hunt",
      "image": "https://...",
      "source": "r6.skin"
    }
  },
  "missing": []
}
```

Runtime lookup will tolerate either object values with an `image` property or direct string values, so the file can stay easy to consume even if it is simplified later.

## Scraper Behavior

The scraper will:

- Query `https://search.r6s.skin/indexes/skins/search` with the bearer token discovered from `/search.js`.
- Search by combinations such as `Silent Hunt Emerald charm`, `Emerald Silent Hunt`, and `ranked charm Silent Hunt Emerald`.
- Normalize rank tiers to `champion`, `diamond`, `emerald`, `platinum`, `gold`, `silver`, `bronze`, and `copper`.
- Skip impossible tiers by season era: no Champion before Ember Rise and no Emerald before Solar Raid.
- Prefer hits that include the tier, season operation name, and charm-like fields or names.
- Write deterministic JSON with sorted keys, a missing list, and a concise terminal summary.
- Support `--dry-run` so the scrape can be audited without changing files.

## Integration

`lib/player.js` will keep `buildRankedCharms()` as the single source of ranked charm generation. Only the image index behind `rankedCharmImage()` changes. This keeps the profile page, shared profile caches, and bulk checker output aligned without adding a second ranked charm code path.

Cached profiles that already contain old fallback badge images will not be rewritten automatically by the runtime change. Existing migration scripts can be adapted later if old cache repair is needed, but fresh checks will use the generated charm image map immediately.

## Error Handling

If the generated file is missing, malformed, or missing a specific tier and season, the app will continue using the existing `skins_cache.json` lookup and rank badge fallback. Scraper network failures will report the failed query and continue collecting other seasons where possible.

The scraper will not scrape at runtime. Runtime profile and bulk checks must remain fast and independent of `r6.skin` availability.

## Testing

Tests will cover:

- Generated ranked charm map entries taking priority over `lib/skins_cache.json`.
- Backward-compatible direct string entries in `data/ranked-charm-images.json`.
- Fallback behavior when a generated entry is missing.
- `buildRankedCharms()` still producing earned tier ladders with real charm images for sample seasons.

Manual verification will run the scraper in dry-run mode, run the focused Node test scripts, and optionally run a small profile or bulk check to confirm rendered ranked charm image URLs now come from the generated map where available.

## Out Of Scope

This work will not redesign the profile UI, change rank history detection, scrape non-ranked charms, or rewrite all existing cached profile files. It only improves ranked charm image data and the shared lookup path.
